기존 웹 프론트엔드 소스코드는 프로젝트 규모에 적합하지 않은 아키텍처, 불필요한 코드중복, 복잡한 로직, 사용하지 않는 코드 등.. 유지보수에 있어 비효율성을 증가시켰고 성능 문제 또한 지속적으로 발생하여 사용자 경험에 부정적인 영향을 미쳤다. 

기술 스택과 아키텍처를 전면 재구축하여 마이크로서비스 기반으로 프로젝트를 재구성 하였다. 

## 리뉴얼 이전에 유지보수 중 겪었던 어려움들

- **공통적으로 사용되는 UI 요소와 로직이 공통화되지 않아 수정 시 과도한 중복 작업 발생**  
    - 예: 차트의 폰트 크기를 변경하려면 30개 이상의 파일을 수정해야 함.
    아젠다 생성 팝업, 분석 팝업 등 여러 UI요소 와 로직들에서도 중복 코드 잔존
        <div className="md:w-[300px]">
        ![image.png](/images/renewal/image.png)
        </div>

- **복잡한 코드 구조와 간결하지않은 가독성 낮은 코드들**
    - 소스코드 이해의 어려움
    - 작은 수정 이슈 처리에도 시간 과다 소요
    - 확장시 구조적 제한에 따른 기능 구현 한계
    - 불필요한 작업 야기
    - 과도한 컴포넌트 분리로 인한 컴포넌트간 데이터 전달을 위한 불필요한 코드들  
    - 버그 발생율 상승

- **사용되지 않은 파일들이 그대로 잔존**
    - 소스코드 탐색시 어려움
    - 소스코드 수정시 혼돈 야기

- **성능저하 및 개발서버(localhost) 속도 저하**
    - 페이지 로드 시간 지연 (blocking time 존재)
    - 불필요한 화면 리렌더링 (+ 불필요한 페이지 전체 새로고침)
    - 개발 환경(localhost)에서 속도 체감 약 3배 저하 (개발속도 저하)
    - 현저히 낮은 페이지 이동간의 딜레이(Interaction to Next Paint - INP)  
    - 화면 로드시 걸리는 시간(Largest Contentful Paint - LCP)  
    - 낮은 개발자 경험 DX  

- **개발 생산성이 낮은 기술스택**
    - 복잡한 Redux 사용: 코드량 과다, boilerplate 증가 (→ react query + zustand)
    - MUI의 쉽지않은 커스터마이징 (→ tailwind css + shadecn UI)
    - 최신화되지 않은 수기로 작성된 api 관련 타입들 (→ swagger to typescript로 타입 자동변환 라이브러리 사용)

## 기존 Monolithic Architecture (모놀리식) 아키텍처의 한계

모놀리식 아키텍처는 초기에는 구현이 간단하고 작은 규모의 프로젝트에 적합할 수 있지만, 규모가 커질수록 한계점들이 따른다.

- 하나의 코드베이스로 구성되므로, 모듈 간 의존성이 높아져 모듈화에 불리. (유지보수가 어렵고 변경 시 예상치 못한 문제 발생)
- 확장성 제한 - 모놀리식에서는 특정 모듈만 확장하려고 해도 전체 애플리케이션을 확장해야 한다.
(특정 기능만 확장이 필요한 경우에도 전체 시스템에 영향을 미친다.)
- 코드 복잡성 증가
- 시간이 지나면서 코드베이스가 커지고 복잡해지며, 모듈 경계가 불명확해질 수 있다.
- 코드가 하나의 프로젝트 안에 섞여 있어 관심사의 분리가 어렵다.
- 하나의 모듈을 수정하면 다른 모듈이나 전체 시스템에도 영향을 줄 가능성이 크다.
- 빌드 시간 및 테스트 시간, 배포 시간이 오래 걸린다. (하나의 서비스만을 배포하여도 앱 전체를 배포해야 한다.)

![image.png](/images/renewal/image%202.png)

마이크로프론트엔드가 프로젝트 규모가 커질수록 유지보수에 유리하다.
